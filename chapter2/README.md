## 2.1 네 개의 영역

DDD 기반 애플리케이션은 다음 **4개의 영역**으로 구성됩니다.

| 영역 | 역할 |
|------|------|
| **표현 (Presentation)** | 사용자 요청을 받아 응용 영역에 전달하고, 결과를 사용자에게 반환 |
| **응용 (Application)** | 사용자에게 제공할 기능을 구현. 도메인 로직을 직접 구현하지 않고 **도메인 모델을 조합**해 기능을 완성 |
| **도메인 (Domain)** | 도메인의 **핵심 로직 구현**. 도메인 모델이 위치하는 곳 |
| **인프라스트럭처 (Infrastructure)** | 구현 기술 담당. DB 연동, 메시지 큐, 외부 API 등 |

```
[표현] 사용자 요청 수신
   ↓
[응용] 기능 흐름 조율 (트랜잭션 단위)
   ↓
[도메인] 핵심 업무 규칙 실행
   ↓
[인프라] DB 저장 / 외부 시스템 연동
```

## 2.2 계층 구조 아키텍처

전통적인 계층 구조에서는 **상위 계층이 하위 계층에 의존**합니다.

```
표현 계층
    ↓ 의존
응용 계층
    ↓ 의존
도메인 계층
    ↓ 의존
인프라 계층
```

- 상위 계층은 하위 계층을 사용할 수 있지만, 하위 계층은 상위 계층을 사용하지 않습니다.
- 유연성을 위해 **응용 계층이 인프라 계층에 직접 의존하지 않도록** DIP를 적용합니다.

## 2.3 DIP — 의존성 역전 원칙

> **고수준 모듈**이 **저수준 모듈**에 직접 의존하지 않도록,  
> 추상화(인터페이스)를 통해 의존 방향을 역전시킵니다.

### 고수준 / 저수준 구분

| 구분 | 설명 | 예시 |
|------|------|------|
| **고수준** | 의미 있는 단일 기능을 제공 (비즈니스 규칙) | "주문이 결제 완료 상태여야 상품을 준비할 수 있다" |
| **저수준** | 고수준의 기능을 구현하기 위한 세부 구현 | SMTP 메일 발송, MySQL 저장, Kafka 발행 |

### 문제 — 직접 의존

```java
// ❌ 응용 계층이 인프라의 구체 클래스에 직접 의존
public class OrderService {
    private MySQLOrderRepository repository; // 구현 클래스 직접 사용
}
```

- `MySQLOrderRepository`가 바뀌면 `OrderService`도 수정해야 합니다.
- 단위 테스트 시 실제 DB가 필요합니다.

### 해결 — 인터페이스 추상화

```java
// ✅ 인터페이스에 의존 — 구현은 인프라 계층에서
public interface OrderRepository { ... }                        // 도메인 계층
public class JpaOrderRepository implements OrderRepository { } // 인프라 계층

public class OrderService {
    private OrderRepository repository; // 인터페이스에만 의존
}
```

의존 방향이 역전됩니다.

```
[응용/도메인] → [<<interface>> OrderRepository] ← [인프라: JpaOrderRepository]
```

### DIP 주의점

DIP를 적용할 때는 **어디서 추상화가 필요한지**를 먼저 판단해야 합니다.

- 단순히 인터페이스를 만드는 것이 목적이 아니라, **고수준 모듈이 저수준 구현에 종속되지 않는 것**이 목표입니다.
- 하위 기능의 변경이 상위 기능에 영향을 주지 않아야 DIP를 올바르게 적용한 것입니다.

## 2.4 도메인 영역의 주요 구성요소

| 구성요소 | 설명 |
|----------|------|
| **엔티티 (Entity)** | 고유 식별자를 가지며 자신의 라이프 사이클을 갖는 객체 |
| **밸류 (Value Object)** | 개념적으로 하나인 값을 표현. 식별자 없음. 불변 설계 |
| **애그리거트 (Aggregate)** | 연관된 엔티티와 밸류를 개념적으로 하나로 묶은 군집 |
| **리포지토리 (Repository)** | 도메인 객체의 영속성 처리. 애그리거트 단위로 존재 |
| **도메인 서비스 (Domain Service)** | 특정 엔티티나 밸류에 속하지 않는 도메인 로직 구현 |

---

## 2.5 애그리거트

> 연관된 객체들을 하나의 군집으로 묶고, 군집의 입구 역할을 하는 **루트 엔티티**를 통해서만 접근한다.

- 복잡한 도메인 객체들을 **하나의 단위로 관리**합니다.
- 외부에서는 **루트 엔티티**를 통해서만 내부 객체에 접근합니다.
- 루트 엔티티가 내부 일관성을 책임집니다.

```
주문 (Aggregate Root: Order)
├── OrderLine (엔티티)
└── ShippingInfo (밸류)
```

Order를 통하지 않고는 OrderLine을 직접 수정할 수 없습니다.  
→ **데이터 일관성 경계** 역할을 합니다.

---

## 2.6 리포지토리

- **애그리거트 단위**로 리포지토리가 존재합니다. (`OrderRepository` O, `OrderLineRepository` X)
- 응용 서비스는 리포지토리에서 애그리거트를 조회해 도메인 로직을 실행합니다.

```java
public class OrderService {
    private OrderRepository orderRepository;

    public void cancel(String orderId) {
        Order order = orderRepository.findById(orderId)
            .orElseThrow();
        order.cancel(); // 도메인 로직 실행
        // 트랜잭션 종료 시 자동 반영
    }
}
```

## 2.7 요청 처리 흐름

```
[사용자]
    │ HTTP 요청
    ▼
[표현 계층 — Controller]
    │ 데이터 변환 (DTO 생성)
    ▼
[응용 계층 — Service / UseCase]
    │ 트랜잭션 관리
    │ 리포지토리에서 애그리거트 조회
    │ 도메인 로직 실행
    │ 결과 반환
    ▼
[도메인 계층 — Entity / VO / Domain Service]
    │ 핵심 업무 규칙 실행
    ▼
[인프라 계층 — Repository Impl / Kafka / 외부 API]
    │ 실제 DB 조회/저장
    ▼
[응답] 사용자에게 반환
```

## 2.8 인프라스트럭처 개요

인프라스트럭처 영역은 **논리적인 개념을 실제로 구현**하는 영역입니다.

- RDBMS 연동 (JPA / QueryDSL)
- 메시지 큐 (Kafka)
- 외부 API 연동 (REST / SOAP)
- 캐시 (Redis)

도메인/응용 계층은 인터페이스에만 의존하고, 실제 구현은 인프라 계층이 담당합니다.  
이 분리 덕분에 **인프라 기술이 교체되더라도 도메인 로직은 영향을 받지 않습니다.**

## 2.9 모듈 구성

모듈은 크게 두 가지 기준으로 나눌 수 있습니다.

**1. 기능 중심 (도메인 중심) 패키지 구성**

```
com.example.order
├── domain
│   ├── Order.java
│   └── OrderRepository.java (interface)
├── application
│   └── OrderService.java
└── infrastructure
    └── JpaOrderRepository.java
```

**2. 계층 중심 패키지 구성**

```
com.example
├── domain
│   ├── order/
│   └── product/
├── application
│   ├── order/
│   └── product/
└── infrastructure
    ├── order/
    └── product/
```

> 어떤 방식이든 중요한 것은, **도메인 로직이 인프라 구현에 의존하지 않는 것**입니다.

### 💬 Topic 1. DIP는 어디까지 적용해야 할까? — 원칙과 실용성 사이

책은 DIP를 통해 고수준 모듈이 저수준 모듈에 의존하지 않도록 해야 한다고 말합니다.  
그런데 현실에서는 이런 상황이 자주 생깁니다.

```java
// 구현체가 하나뿐인데 인터페이스를 만들어야 할까?
public interface GetOutboundUseCase {
    OutboundDto invoke(Long outboundId);
}
public class GetOutboundUseCaseImpl implements GetOutboundUseCase { ... }
```

인터페이스를 만드는 비용(파일 증가, 네비게이션 불편)이 얻는 이점보다 클 수 있습니다.

**생각해볼 것:**
- "구현체가 하나뿐인 인터페이스"는 DIP를 올바르게 적용한 것인가, 오버엔지니어링인가?
- 언제는 인터페이스가 필요하지않고 언제는 필요한가? 기준을 어떻게 세울 수 있을까?

### 💬 Topic 2. 애그리거트 경계는 어떻게 설정해야 할까?

책은 "연관된 객체를 하나의 군집으로 묶어라"고 하지만, **경계를 어디서 긋느냐**는 여전히 어렵습니다.

```
// 경계를 크게 잡으면
주문(Order)
├── OrderLine (N개)
├── ShippingInfo
└── Payment         ← 결제도 주문에 포함?

// 경계를 작게 잡으면
주문(Order) / 결제(Payment) 를 별도 애그리거트로
```

크게 잡으면 → 하나의 트랜잭션에서 너무 많은 것을 잠급니다. (동시성 문제, 성능 저하)  
작게 잡으면 → 두 애그리거트 간의 일관성을 어떻게 보장할지 고민이 생깁니다.

**생각해볼 것:**
- 애그리거트가 너무 클 때 발생하는 실제 문제를 경험해봤는가?
